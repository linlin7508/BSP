# 📌 Mix Zone 規則（Revised Step-by-Step Protocol）

---

# 🧠 Step 0 — System Trigger Initialization

```text id="s0"
When EV enters mzTrigger radius (e.g., 150–220m),
RSU initializes a new Mix Zone Session.
```

RSU actions：

* start session
* start candidate collection
* initialize empty membership set

---

# 🧠 Step 1 — Candidate Collection (Static Snapshot)

```text id="s1"
Collect all vehicles within candidateRadius at session start.
```

Rules：

* only vehicles detected at initialization time are included
* no late join allowed (static membership)

```text id="code1"
session.members = snapshot(candidateRadius vehicles)
session.frozen = true
```

---

# 🧠 Step 2 — Communication Readiness Check

```text id="s2"
Each member reports COMM_READY status.
```

Definition：

* vehicle is reachable via V2X
* RSU receives ACK

```text id="code2"
ready_set = {v ∈ session.members | commOK(v)}
```

---

# 🧠 Step 3 — Quorum Decision (90% Rule + Timeout)

```text id="s3"
If |ready_set| ≥ 0.9 × |session.members|
OR timeout exceeded → trigger BSP.
```

Important：

* timeout ensures liveness
* TTC NOT included

```text id="code3"
if (ready_ratio >= 0.9 || timeout_3s) {
    startBSP();
}
```

---

# 🧠 Step 4 — Synchronized BSP Execution

```text id="s4"
RSU broadcasts:
Tb (Broadcast Phase): 4 seconds
Ts (Silence Phase): 1 second
```

Flow：

1. RSU broadcasts pseudonym change command
2. Vehicles enter silent phase (Ts)
3. At synchronized timestamp → all change pseudonym

```text id="code4"
CMD_CHANGE_PSEUDONYM at T_sync
```

---

# 🧠 Step 5 — Pseudonym Update (Atomic Change)

```text id="s5"
All session members update pseudonym simultaneously.
```

Guarantee：

* same timestamp
* no incremental or asynchronous update

---

# 🧠 Step 6 — Post-Mix Zone Retention (EV-driven)

```text id="s6"
After EV exits Mix Zone,
vehicles maintain current pseudonym for 20 seconds
or until leaving zone.
```

Important：

* no new vehicles are added
* no new BSP triggered

```text id="code5"
pseudonym_valid_until = EV_exit_time + 20s
```

---

# 🧠 Step 7 — Session Termination

```text id="s7"
Session ends when:
- EV leaves zone AND
- retention period expires
```

RSU cleanup：

* clear session.members
* reset state machine

## 問題修正詳解

---

### 🔴 問題 1 — WAIT_READY → BSP_ACTIVE 必須 Event-Driven

**錯誤設計（polling）：**
```cpp
// ❌ 每個 tick 都 check，timing jitter，log 不可重現
if (commReadyCount_ >= 0.9 * frozenMemberCount_ || simTime() >= commReadyDeadline_) {
    startBspPhase();
}
```

**問題根源：** OMNeT++ 的 tick-based polling 導致轉換時刻不確定；不同 simulation run 可能在不同 tick 觸發。

**正確修法：Event-Driven Hybrid Timer**

**Step A — RSU 新增 `commReadyTimer_`：**
```cpp
// rsu.h 新增
omnetpp::cMessage* commReadyTimer_ = nullptr;
```

**Step B — `startMixZone()` 排程 deterministic timeout：**
```cpp
// 取消舊 timer（防止重入）
if (commReadyTimer_ && commReadyTimer_->isScheduled())
    cancelEvent(commReadyTimer_);
commReadyTimer_ = new omnetpp::cMessage("commReadyTimeout");
scheduleAt(commReadyDeadline_, commReadyTimer_);
```

**Step C — `handleSelfMsg()` 處理 timeout（deterministic path）：**
```cpp
if (msg == commReadyTimer_) {
    commReadyTimer_ = nullptr;
    EV_INFO << "[RSU " << rsuIndex_ << "] COMM_READY timeout at t="
            << simTime() << ", count=" << commReadyCount_
            << "/" << frozenMemberCount_ << "\n";
    triggerBspPhase();   // 確定性觸發，不依賴 tick 位置
    return;
}
```

**Step D — `onWSM()` 達到 quorum 時取消 timer（另一條 deterministic path）：**
```cpp
if (mzState_ == MzState::WAIT_READY && !isLeader) {
    if (/* 收到 AckBspMsg */ && activeSession_.members.count(vid)) {
        ackedMembers_.insert(vid);
        if ((int)ackedMembers_.size() >= (int)(0.9 * frozenMemberCount_)) {
            if (commReadyTimer_ && commReadyTimer_->isScheduled())
                cancelAndDelete(commReadyTimer_);
            commReadyTimer_ = nullptr;
            triggerBspPhase();   // quorum 路徑，同樣確定性
        }
    }
}
```

> ✅ 結果：**Deterministic Hybrid Event System**
> - Quorum path：收到第 N 個 ACK 訊息那一刻觸發（精確）
> - Timeout path：scheduleAt 保證 simTime 精確（不漂移）
> - 任何 run 都可重現

### 🔴 問題 2 — ACK 必須是獨立訊息，非 State Proxy

**錯誤設計（flag + BSM 帶出）：**
```cpp
// ❌ ACK 發生在 RSU 指令之後，非因果獨立
hasAckedBsp_ = true;
hw->setHasAckedBsp(hasAckedBsp_);  // 跟著 BSM 週期走，非獨立回應
```

**問題：** ACK 語義應代表「車輛收到指令且確認運作正常」；透過 BSM 帶出會有最長 100ms 的延遲（BSM 週期），且 RSU 無法區分「上一個 tick 的 BSM 帶的 flag」還是「新鮮回應」。

**正確修法：新增獨立 `AckBspMsg` 訊息**

**新訊息定義（新建 `AckBspMsg.msg` 或加入 `MixzoneMsg.msg`）：**
```
// 加入 MixzoneMsgType enum（MixzoneMsg.msg）
COMM_READY = 10,   // Vehicle → RSU 的獨立 ACK
```

**車端 `onWSM` 收到 `CMD_START_BSP` 後立即回傳：**
```cpp
case MixzoneMsgType::CMD_START_BSP: {
    if (vehMzState_ == VehMzState::NORMAL || vehMzState_ == VehMzState::IN_MZ_WAIT) {
        mySessionId_ = mz->getSessionId();
        // ... 設定 zoneCenter_, zoneRadius_, tbUntil_, silenceUntil_ ...
        transitionTo(VehMzState::IN_BSP);

        // ✅ 立即回傳獨立 ACK 訊息（因果乾淨）
        auto* ack = new MixzoneMsg("COMM_READY");
        ack->setMsgType(MixzoneMsgType::COMM_READY);
        ack->setSessionId(mySessionId_);
        setTargetVehIndexForAck(ack, vehIndex_);   // 或直接用欄位
        populateWSM(ack);
        sendDown(ack);

        writeEvent("BSP_START_ACK_SENT");
    }
    break;
}
```

**RSU `onWSM` 收到 COMM_READY 訊息（非 BSM）：**
```cpp
// 注意：現在 onWSM 可能收到 MixzoneMsg，而非只有 Homework
// 需要在 onWSM 開頭先嘗試轉型 MixzoneMsg
MixzoneMsg* mzAck = dynamic_cast<MixzoneMsg*>(wsm);
if (!mzAck) mzAck = dynamic_cast<MixzoneMsg*>(wsm->getEncapsulatedPacket());

if (mzAck && mzAck->getMsgType() == MixzoneMsgType::COMM_READY) {
    if (mzState_ == MzState::WAIT_READY &&
        mzAck->getSessionId() == activeSession_.id) {
        int ackVid = mzAck->getTargetVehIndex();  // 車端填自己的 vid
        if (activeSession_.members.count(ackVid)) {
            ackedMembers_.insert(ackVid);
            if ((int)ackedMembers_.size() >= (int)(0.9 * frozenMemberCount_)) {
                cancelAndDelete(commReadyTimer_);
                commReadyTimer_ = nullptr;
                triggerBspPhase();
            }
        }
    }
    return;   // 不繼續處理為 Homework BSM
}
```

> ✅ ACK 是獨立 MixzoneMsg，到達 RSU 即刻計數，因果鏈清晰：
> `CMD_START_BSP(RSU→Veh)` → `COMM_READY(Veh→RSU)` → `triggerBspPhase()`

### 🟡 問題 3 — Frozen Snapshot 的 Determinism 保護

**Edge Case：** 若車輛在與 `evaluateTrigger()` 完全相同的 simtime tick 進入範圍，
mobility 更新順序可能導致一次 run 包含該車、另一次 run 不包含。

**修正：在 `startMixZone()` 記錄凍結時刻，後續同 tick 的新車一律拒絕：**

```cpp
void rsu::startMixZone(int evVid, double evDist,
                       const std::unordered_set<int>& members)
{
    frozenSnapTime_    = simTime();          // 凍結時刻
    frozenMemberCount_ = (int)members.size();
    activeSession_.members = members;        // 同一行立即凍結

    // 後續 Mid-way Join Prevention 已有（onWSM 對非成員 Yield）
    // 額外：evaluateTrigger 結束後不再 append 新成員
    // ...
}
```

**`evaluateTrigger()` 加防重入保護：**
```cpp
bool rsu::evaluateTrigger()
{
    // 在同一 simtime tick 內不重複觸發
    if (mzActive_) return false;
    // ...
}
```

> OMNeT++ 的離散事件保證同一 simtime tick 內事件序列固定，
> `startMixZone()` 一旦呼叫，`mzActive_ = true`，同 tick 後續事件無法再觸發。

---

### 🟡 問題 4 — logPrivacyMetrics 唯一觸發點

**問題：** quorum path 和 fallback path 都呼叫 `logPrivacyMetrics()`，可能 duplicate。

**正確設計：** Privacy metrics 是 AS 的屬性，與執行路徑無關。

**唯一觸發點 = `triggerBspPhase()`：**
```cpp
void rsu::triggerBspPhase()
{
    if (mzState_ != MzState::WAIT_READY) return;  // 防重入

    // ✅ 唯一觸發點：此時 AS 已確定，log 一次
    logPrivacyMetrics(activeSession_.id, frozenMemberCount_);

    mzState_ = MzState::BSP_ACTIVE;
    tbStart_  = simTime();
    tbEnd_    = tbStart_ + bspBroadcast_;
    tsStart_  = tbEnd_;
    tsEnd_    = tsStart_ + bspSilence_;
    if (mzEnd_ > SIMTIME_ZERO && mzEnd_ < tsEnd_) tsEnd_ = mzEnd_;

    // 廣播 CMD_START_BSP 給所有凍結成員
    for (int vid : activeSession_.members) {
        auto m = new MixzoneMsg("CMD_START_BSP");
        // ... 填欄位 ...
        sendDown(m);
    }
    writeEvent("BSP_PHASE_STARTED", frozenMemberCount_, pendingEv_.evDist,
               activeSession_.id, commReadyCount_, frozenMemberCount_, 0);
}
```

> ✅ 無論走 quorum path 或 timeout path，都只在進入 `triggerBspPhase()` 時記錄一次。
> quorum 達成後送 `CMD_CHANGE_PSEUDONYM`（SILENCE → 換假名）不重複記錄。

### 🟡 問題 5 — TTC：回答「為何收集？」

**論文回應方式：**

> *"TTC is computed continuously for every vehicle and logged to the safety evaluation CSV. This allows post-hoc analysis of whether the 1-second silence window (Ts) falls within a safe TTC margin, validating the 90% safety threshold without coupling the safety metric to the protocol control path."*

**實作：** `checkSafetyForSilence()` 保留完整 TTC/TTS 計算，log 到 CSV，返回 `true`。
**驗證：** 模擬後分析 `P(TTC > Ts) = P(TTC > 1s)`，此即安全率。

## RSU 完整事件流程（更新版）

```
t=0  EV enters mzTrigger
        evaluateTrigger() → startMixZone()
        frozenMemberCount_ = N
        scheduleAt(t+4s, commReadyTimer_)
        mzState_ → WAIT_READY

t=0~4s  Vehicles receive CMD_START_BSP (broadcast all N members)
        Each vehicle → transitionTo(IN_BSP) → sendDown(COMM_READY msg)

t=Xack  RSU onWSM receives COMM_READY from vid k
        ackedMembers_.insert(k)
        if size >= 0.9*N:
            cancelAndDelete(commReadyTimer_)
            triggerBspPhase()           ← quorum path (deterministic)
                logPrivacyMetrics()     ← 唯一觸發點
                mzState_ → BSP_ACTIVE
                tbStart_ = now, tbEnd_ = now+4s

t=t0+4s (OR timeout)
        commReadyTimer_ fires → triggerBspPhase()  ← timeout path

t=tbEnd_
        mzState_ → SILENCE
        broadcast CMD_START_SILENCE to all N members

t=tsEnd_ (OR quorum via BSM vehMzState==PSEUDONYM_PENDING)
        broadcast CMD_CHANGE_PSEUDONYM
        mzState_ → DRAINING

t=EV leaves OR allVehiclesFinished()
        deactivateMixZone() → RESUME_BROADCAST
        full reset
        mzState_ → INACTIVE
```

# Walkthrough - Static Closed-Set Mix Zone Implementation

We have successfully refactored the Mix Zone protocol into a **Static Closed-Set Experiment** model. This ensures that privacy metrics (Success Rate, Entropy) are calculated based on a stable, locked anonymity set (AS), providing accurate academic evaluation.

## Key Improvements

### 1. Candidate Freeze & Snapshot Invariants
The Anonymity Set is now frozen at the exact moment of the RSU trigger.
- **Frozen AS**: Once a session starts, the membership list is immutable. New vehicles entering the range are rejected from the session.
- **Deterministic Metrics**: Success Rate ($1/|AS|$) and Entropy ($\log_2|AS|$) are recorded exactly once at the RSU, at the transition from setup to active broadcast.

### 2. Event-Driven Quorum Mechanism
We replaced the polling-based state machine with a **Deterministic Hybrid Event System**:
- **`COMM_READY` Messages**: Vehicles now send an explicit ACK message back to the RSU upon receiving the BSP command.
- **`commReadyTimer_`**: The RSU uses a specific `omnetpp::cMessage` timer to trigger the BSP phase either when the 90% quorum is reached OR when a 4-second window expires. This eliminates timing jitter and ensures reproducible simulation runs.

### 3. Evaluation-Only Safety Metrics
Safety logic (TTC/TTS) is now decoupled from protocol flow:
- `checkSafetyForSilence()` always returns `true`, ensuring that pseudonym rotation is NOT blocked.
- Collision risks are still calculated and logged to the CSV for post-hoc safety analysis (confirming the $P(TTC > T_s) \ge 0.9$ requirement).

### 4. Privacy Metrics Output
A new output file `results/privacy_metrics.csv` is generated, containing:
- `session_id`: Unique experiment ID.
- `timestamp`: Time of AS confirmation.
- `AS_size`: Number of frozen members.
- `SR`: Success Rate ($1/N$).
- `entropy`: Anonymity set entropy.

## Technical Details

### RSU State Machine (5 States)
1. `INACTIVE`: Idle state.
2. `WAIT_READY`: AS is frozen; waiting for `COMM_READY` ACKs.
3. `BSP_ACTIVE`: Broadcast ($T_b$) phase.
4. `SILENCE`: Silence ($T_s$) phase; tracking quorum for rotation.
5. `DRAINING`: Session complete; waiting for vehicles to exit.

### Message Sequence
1. **RSU** → `CMD_START_BSP` (Freeze AS)
2. **Vehicle** → `COMM_READY` (ACK signal)
3. **RSU** → `triggerBspPhase()` (Start Broadcast & Log Privacy)
4. **RSU** → `CMD_START_SILENCE` (Start Silence Phase)
5. **RSU** → `CMD_CHANGE_PSEUDONYM` (Rotated Identity)

- **Log Integrity**: Events like `START_MZ_WAIT_READY` and `BSP_PHASE_STARTED` accurately capture the causal chain of the protocol.
